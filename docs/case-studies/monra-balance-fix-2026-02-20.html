What Happened

We had a detailed implementation plan — stuck transactions, balance drift, incorrect frontend classification across Monra's ledger system. Five tasks. Four files. Clear root cause analysis.

Claude Code went straight to implementation without querying Engram. No memory. No prior context. Start from scratch. Exactly what every agent does today.

After the session closed, we ran three retroactive Engram searches — measuring what would have been found if the agent had looked.

What Engram Found

Query 1 — "balance fix attribution"
Feb 7 session: balance correct, amounts stale
Mentions attributeTransaction needing destinationAmount. Directly relevant to the balance/transaction interaction patterns — exactly the pattern we were fixing.
Feb 7 session balance patterns
Query 2 — "transfer link handlers escrow"
Feb 16-17: Full dependency chain the plan missed
These sessions contained the complete picture: no webhook handler for escrow deposits, legacy balance check gaps, the naming confusion between transfer_link_created vs escrow_lock, and linkTransferBroadcastSuccess end-to-end wiring listed as pending — exactly what we fixed today. The plan didn't mention CDK, API gateway, types, or schemas. These sessions showed all of it.
Feb 16 session Feb 17 session escrow architecture CDK dependencies
Query 3 — "CDK stack Lambda deploy"
CDK deletion incident — live Lambdas at risk
A CloudFormation incident deleting live Lambdas was documented in past sessions. No prior CDK deployment context existed in Engram before this session — confirming: if Engram had been running during Feb 17 when those Lambdas were first created, we would have known about the deletion risk before touching the stack.

The planned tasks themselves benefited minimally — the plan was specific, the agent could read files directly. The value was entirely in what the plan didn't know existed.

Key Finding
Engram's highest value is surfacing implicit dependencies that aren't in the plan — because the planner didn't know they existed.
04

The Problem This Proves

The agent skipped Engram voluntarily. The recall step was in the prompt. It was optional. So it was skipped.

Every AI agent today has the same amnesia. Plans miss what the planner doesn't know. Sessions end and everything is lost. The next agent starts from zero, re-discovers the same codebase, hits the same implicit dependencies, burns the same tokens.

The Fix — Day 6 of Execution Plan
SessionStart auto-injection. Before the agent makes its first tool call, Engram automatically queries past sessions based on current project context and injects the results. The agent can't skip what it never chose to do.
