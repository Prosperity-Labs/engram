# Experiment 002: Context Injection — Real Feature Task

> Date: 2026-02-22
> Status: Complete (Sessions A + B + C)

## Hypothesis

Auto-generated engram brief (file stats, co-change clusters, danger zones) reduces exploration time when implementing a real feature. Additionally: can a cached file summary prevent the agent from reading the file entirely?

## Task

> "Implement Alchemy webhooks for escrow contract address activity to eliminate the broadcast-success race condition."

Real monra-app feature with:
- A specific bug (race condition in broadcast-success)
- A known file with history (alchemy-webhook.ts — 27 writes across 12 sessions)
- A clear success criterion (webhook becomes source of truth)

## Setup

| Session | Branch | CLAUDE.md content | Hooks |
|---------|--------|-------------------|-------|
| A (cold) | `experiment/webhook-cold` | None | Removed |
| B (engram brief) | `experiment/webhook-context` | Auto-generated engram slim brief | Removed |
| C (cached summary) | `experiment/webhook-cached-summary` | Engram brief + **hand-written** file summary of alchemy-webhook.ts | Removed |

Session B's CLAUDE.md was auto-generated by `engram brief --slim`. Session C added a hand-written cached summary on top — created by reading the actual 594-line file and compressing it to ~200 words covering purpose, exports, routing logic, race condition handling, idempotency, and dependencies. Included the instruction "Do not re-read this file — use this summary directly."

## Analysis

### What worked

1. **Engram brief cuts planning time ~55%** — consistent across B (7.6 min) and C (7.4 min) vs A (17.2 min). The Key Files list guides the agent to important files faster.
2. **Fewer blind exploration subagents** — 3 in B and C vs 5 in A. The brief reduces "let me search broadly" behavior.
3. **Target file found sooner** — #5 and #6 vs #8. The brief listing `alchemy-webhook.ts — 15 edits` signals its importance.
4. **Session C used 53% fewer input tokens** (15,982 vs 33,961). The cached summary gave the model a head start — less processing needed even though it still read the file.
5. **Session C found the plan doc first** — the cached summary's architectural context may have oriented the agent toward documentation before code.

### What didn't work

1. **Cached summary did NOT prevent the file read.** The model read alchemy-webhook.ts in all three sessions despite a detailed 200-word summary and explicit "do not re-read" instruction. Models want to see actual code.
2. **FUSE virtual filesystem approach is not worth building.** The core assumption (cached summaries replace reads) failed.
3. **Co-change clusters were not acted on.** `validators.ts` was never proactively read in any of the three sessions, despite the brief listing `validators.ts + handlers.ts` as a co-change cluster.
4. **Key Decisions section is noise.** The auto-generated entries include user chat messages ("Thanks! Let's /compact"), system prompts ("Entered plan mode"), and only one actual decision. The `_architecture_patterns()` keyword search matches boilerplate, not real decisions. None of the three sessions appeared to use anything from this section.
5. **Danger Zones were irrelevant.** The flagged files (BalanceService.ts, bridge.ts) weren't involved in this task. Danger zones are task-agnostic — they warn about files that aren't being touched.

### Confounding variable: Cache warmth

Sessions ran sequentially on the same codebase. Cache read tokens: A=1.6M, B=2.2M, C=2.0M. The 600k jump from A→B may include server-side prompt cache hits from Session A's API calls, not just the engram brief. The speedup is real but partially confounded. Clean validation requires either:
- Docker isolation with cache TTL gaps (5+ min between sessions)
- External tester on an unrelated codebase

## Conclusions

### Validated
- **File stats in CLAUDE.md guide navigation.** Even shallow "X edits, Y sessions" helps the agent prioritize which files to read first. This is engram's strongest signal.
- **Slim brief is the right format.** <500 tokens, always-on, low cost. The agent uses it.
- **Consistent ~55% speedup** on planning phase (n=2, cache-confounded, but directionally strong).

### Invalidated
- **Cached file summaries don't prevent reads.** The model reads source code regardless of summaries. FUSE filesystem approach is not viable.
- **"Do not re-read" instructions are ignored.** The model treats CLAUDE.md as advisory, not authoritative for tool decisions.

### Needs improvement
- **Key Decisions:** Filter to role=assistant, min content length, exclude boilerplate ("plan mode", "/compact", system prompts).
- **Co-change clusters:** The agent doesn't act on them. Either the format needs to be more directive ("ALWAYS edit validators.ts when touching handlers.ts") or this section isn't useful.
- **Danger Zones:** Task-agnostic warnings are low-signal. Consider task-aware danger detection.
- **Architectural context:** The gap between "file stats" and "design intent" is where the real value lives. Engram can generate the former but not yet the latter.

## What to build next

1. **Fix Key Decisions** — filter noise, extract real architectural decisions only
2. **Skip FUSE/caching approach** — models read source code, don't try to prevent it
3. **Focus on navigation speed** — the brief's file list is the proven value. Make it better: include file paths (not just names), add one-line purpose descriptions
4. **Test on external codebase** — eliminate cache confound with a clean environment
5. **Consider PreToolUse hook value** — the hook injects per-file context at Read time. Given that the model reads files regardless, injecting "last error on this file" at read time may be more valuable than trying to prevent the read
