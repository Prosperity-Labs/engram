# Experiment 002: Context Injection — Real Feature Task

> Date: 2026-02-22
> Status: Complete (Session A + B), Session C planned

## Hypothesis

Auto-generated engram brief (file stats, co-change clusters, danger zones) reduces exploration time when implementing a real feature, even without hand-crafted architectural context.

## Task

> "Implement Alchemy webhooks for escrow contract address activity to eliminate the broadcast-success race condition."

Real monra-app feature with:
- A specific bug (race condition in broadcast-success)
- A known file with history (alchemy-webhook.ts — 27 writes across 12 sessions)
- A clear success criterion (webhook becomes source of truth)

## Analysis

### Confounding Variable: Cache Warmth

**Important caveat:** Session B ran after Session A on the same repo. Claude Code's cache read jumped from 1.6M to 2.2M tokens — 600k tokens of cached context from previous sessions. Some of the speed improvement may be cache hits from Session A's exploration, not the engram brief.

The branches were isolated (`experiment/webhook-cold` vs `experiment/webhook-context`) but the model's cache was not. This is inherent to running sequential experiments on the same codebase with the same Claude Code instance.

**Honest conclusion:** 55% faster on a warm codebase with engram brief vs cold session. Controls for cache warmth needed before making strong claims. The cleanest validation would be running this on an external contributor's codebase — someone whose sessions have never touched this code.

### What the engram brief likely helped with

1. **55% faster** — 7.6 min vs 17.2 min, but partially confounded by cache warmth
2. **Target file found sooner** — tool call #5 vs #8. The brief listed `alchemy-webhook.ts` as a key file, likely guiding the agent to read it first
3. **Fewer exploration subagents** — 3 vs 5 Task spawns. The brief's file list reduced blind discovery
4. **Found plan doc** — Session B found `escrow-link-transfer-implementation-plan.md` which Session A missed. The brief may have signaled that related docs exist

### What the brief did NOT help with

1. **No reduction in total tool calls** — 23 vs 19. More targeted work, not less work
2. **Co-change awareness was partial** — did not proactively touch `validators.ts` despite the co-change cluster listing
3. **Key Decisions section was noise** — "Thanks! Let's /compact" is not a useful decision. Needs filtering
4. **Danger Zones were irrelevant** — task didn't involve the flagged files

### What this means for engram

The auto-generated brief is **useful but shallow**. It tells the agent *where* things are (file stats reduce navigation time) but not *what* matters about them (architectural knowledge, known bugs, design intent).

Observable improvements likely came from:
- **Key Files list** — agent knew which files were important before starting
- **Reduced exploration uncertainty** — fewer blind exploration subagents

No observable impact from:
- Danger zone warnings (irrelevant to this task)
- Co-change clusters (agent didn't act on them)
- Key Decisions (noise, not signal)

## Experiment C: Cached File Summary (Planned)

**Hypothesis:** If the brief includes a cached function summary for alchemy-webhook.ts, the agent will skip reading the file entirely.

**Setup:** Add to CLAUDE.md:
```markdown
## alchemy-webhook.ts (cached summary)
Exports: processAlchemyWebhook(payload)
Pattern: matches txHash to pending transfers by amount/sender
Location: monra-core/flow-service/src/listeners/alchemy-webhook.ts
Do not re-read this file — use this summary directly.
```

**Measuring:** Does the agent skip reading alchemy-webhook.ts?
- If yes → cached summaries work, virtual filesystem layer is worth building
- If no → the model reads anyway, caching at this level is pointless

## Implications for Engram Roadmap

1. **File stats work** — even shallow "X edits, Y sessions" guides the agent. Keep this in slim brief.
2. **Key Decisions needs filtering** — current keyword search pulls garbage. Filter to role=assistant, min length, exclude boilerplate.
3. **Architectural context is the gap** — the jump from "file stats" to "design intent" is where the real value lives. This is what hand-crafted context would provide that auto-generation doesn't yet.
4. **Cached summaries (Experiment C)** — if this works, it opens a path to a FUSE/virtual layer that serves file summaries instead of full content, saving massive token cost.
